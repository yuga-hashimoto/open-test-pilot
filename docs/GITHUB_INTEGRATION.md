# GitHub Integration

GitHub OAuth is the only user authentication method. A GitHub App supplies short-lived installation access for clone/fetch, push, webhooks, Pull Requests, Checks, Commit Status, and PR comments. The adapter signs the App JWT with the configured private key, exchanges it for an installation token, and never returns that token from the server's OAuth callback. The server treats GitHub as the source repository; Claude Code uploads are proposals or artifacts, not a replacement source of truth.

The current foundation includes `@open-test-pilot/github-adapter` with OAuth authorization URL construction, code exchange, GitHub App JWT signing, installation-token exchange, repository lookup, branch/commit/PR/check/status/comment operations, and timing-safe `X-Hub-Signature-256` verification. Credentials are supplied through deployment configuration; tokens are never written to Manifests or returned in logs.

For a local GitHub App installation, configure `GITHUB_APP_ID`, `GITHUB_INSTALLATION_ID`, and `GITHUB_PRIVATE_KEY_PATH`. Keep the downloaded PEM outside version control; this repository ignores `.secrets/`. The reference App uses repository permissions for Checks, Commit statuses, Contents, Issues, and Pull requests, plus mandatory Metadata read access. Installation tokens are short-lived and must be requested at runtime rather than stored.

The local Server exposes `GET /auth/github/start` and `GET /auth/github/callback`. It stores one-time OAuth state, resolves the GitHub user, and returns a short-lived OpenTestPilot session token after a successful code exchange; it never returns the GitHub access token to the browser response. The Web UI's GitHub login screen starts this flow, completes the callback at `/auth/github/callback`, keeps only the OpenTestPilot session token, and then loads or creates an organization through `GET /v1/me/organizations` and `POST /v1/organizations`. Set `AUTH_REQUIRED=true` in hosted mode to require `Authorization: Bearer <sessionToken>` plus organization membership on `/v1` routes. Set `VITE_OPENTESTPILOT_URL` and `VITE_OPENTESTPILOT_AUTH_REQUIRED=true` for the hosted Web UI; an organization can be supplied with `VITE_OPENTESTPILOT_ORGANIZATION_ID` or selected after login. Local development can keep the explicit `x-organization-id` tenant header path.

The default edit path creates a dedicated branch and Pull Request. Protected branches cannot receive direct commits. Repository permission loss revokes Platform project access. Webhook signatures, installation ownership, branch/PR state, and commit SHA are persisted and audited. A verified `push`, `pull_request`, `create`, or `branch_protection_rule` delivery for a linked repository enqueues every enabled tenant schedule and returns the resulting run IDs; delivery IDs are idempotent, so a replay does not enqueue a second run. Repository content synchronization still requires the configured App installation and is kept separate from webhook acknowledgement.

On 2026-07-17 the configured installation was verified against `yuga-hashimoto/open-test-pilot` using the real PEM-backed App flow. The server completed sync, branch listing, content read, temporary branch creation, file commit, read-back, compare, Draft PR creation, PR history, Check Run, Commit Status, and cleanup; temporary Draft PRs `#2` and `#3` were closed and their branches deleted. The run notification route also created Check Run `87859778091` and a success status on `main`. The OAuth browser flow is intentionally separate: it requires a configured OAuth client ID/secret and GitHub sudo-mode approval, and no OAuth secret is stored in the repository.
